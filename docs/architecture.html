核心闭环：日志源 → 降噪 → 编排 → 根因分类 → (代码/非代码分支) → 方案+置信度 → 找最懂的人 → 邮件/IM 通知
本版不含：PR 自动化（后置）、SCM Provider 双平台适配（后置）
关键时序点：
1. 降噪在 LLM 之前，避免成本爆炸与噪音。
2. 根因类型先分类，代码 bug 才进 repo 检索，非代码走轻量分析。
3. 两阶段 LLM（候选到验证）显著提升根因准确率，对冲 OpenRCA 指出的难点。
4. 置信度贯穿全程，低置信走"辅助 triage"而非自动定责。
5. 反馈闭环：负责人对根因/方案的评价回灌 RAG 知识库，持续提升。
共享 Repo 检索：多手段配合,非单一RAG。grep精确检索(找符号/引用/字面量) + codegraph结构检索(调用图/依赖图/数据流) + 向量语义检索(模糊匹配) + 历史案例RAG。LLM agent 自主决定用哪种手段、多轮迭代检索,类似 Copilot 检索代码的范式。代码与非代码分支都调用这套能力。
新增"缺文件/资源"分支：先查代码中的路径引用/资源加载点，确认该文件是外部依赖还是应内置，再判定是外部文件缺失还是代码缺失。
两阶段 LLM：阶段1 生成候选根因，阶段2 用实际代码上下文验证/否决，避免单轮"瞎猜"。
置信度分流：高置信自动定责+带方案；低置信退化为辅助 triage（给候选+人选，让人判定），保护信任。
核心定位: 把分析结果变成"给最懂代码的人的一份可执行修复请求"。产品姿态是"修复助手"不是"问责工具",团队接受度天差地别。
Owner 四级 fallback: CODEOWNERS(精确,团队维护) 到 OWNER文件(半结构化) 到 commit blame(出错方法主要贡献者,非最后一行作者) 到 README(LLM读兜底)。逐级 fallback 保证总能找到人。输出含来源标记("为什么找你")。
blame 改进: 不取出错行最后作者(可能只改了typo),取出错方法/类的主要贡献者。避免大重构自动修改导致的误判。
多平台身份映射: 邮箱作统一主键, GitHub username / ADO uniqueName 各自映射, 跨平台找到同一个人。
通知四要素: 根因摘要(位置+代码+为什么+类型) + 修改方案(代码bug给diff/非代码给建议动作) + 证据链(stack+变量+调用链+历史) + 置信度(高/中/低+依据)。让他收到能立即行动。
通知编排: 去重(同人短时合并) + 升级(高优N分钟未响应升oncall) + 静默时段(非紧急不凌晨发) + 频率控制(每人每日上限,超了聚合)。避免轰炸。
反馈闭环(飞轮): 四类反馈 - 根因准不准/方案能不能用/找错人没/已知问题。用途: 优化模型(采纳=强正信号) + 更新owner映射 + 加白名单 + 存RAG知识库。系统越用越准。
方案是起点非终点: 他最熟, 给起点比从零排查快。可改后采纳或否决自己修。反馈"改后采纳"是强优化信号。
成本漏斗：原始日志 10万条/分 - 多行聚合 - ERROR过滤 - 指纹聚合 - 时间窗去重 - 预过滤 - 进 LLM 仅 2-3 个/分。这一层做不好,整个系统因成本和噪音无法落地。
级别过滤可配置：默认 ERROR 及以上,但支持按服务/环境单独配置。有些团队业务异常打 WARN,有些服务 FATAL 才需关注,不能一刀切。配置项: 全局默认阈值 + 按服务覆盖 + 按环境覆盖(如 prod 比 staging 更宽松)。
指纹算法：不能 hash 整条日志(时间戳/host不同)。取 stack top-N 帧 + 异常类型 + message 模板,归一化动态段后 hash。偏粗会漏,偏细会炸,建议保守上线靠反馈调。
升级检测：静默期内若 host 数突增或堆栈变深,视为升级,突破 cooldown 重新触发分析。
预过滤三分流：白名单丢弃、规则/小模型判定非代码的直接分流、拿不准的才送 LLM。
保留原始样本：聚合后仍保留 1-2 条完整原始日志供 LLM 引用,否则 LLM 拿不到细节。
可回放：被过滤的异常可事后回查,避免过滤太狠漏真 bug 无法追溯。
核心职责:让重要异常优先分析,不重要延后或降级,LLM调用始终在可控预算内。无此层,洪峰时要么烧爆预算、要么雪崩卡死、要么关键事故被低优事件淹没。
优先级信号:影响范围+严重度+频率趋势+业务影响+新鲜度,加权打分分 P0-P3 四级。事件在队列等待时若影响扩大,优先级动态上调。
去抖 vs 降噪去重:降噪层去重基于 fingerprint 相同;编排层去抖针对同一根因的不同表象(发布 regression 触发多个不同 fingerprint),聚批后一次分析。P0 跳过去抖立即分析。
限流三维度:全局并发+QPS+token预算,按优先级分配配额。P0 占六成并发保证关键事故不被挤占。
降级关键原则:通知不能断。即使 LLM 挂了,至少把"有异常+找到的人+原始日志"通知出去让人工接手。分析能力可降级,告警能力必须高可用。
部分成功:根因分析成功但方案生成失败,根因仍通知,不浪费已有分析成果。
熔断:LLM 连续失败超阈值触发熔断,暂停调用一段时间,避免雪崩。
核心挑战: OpenRCA(ICLR2025)证明LLM定位代码根因准确率仍低。这层设计核心不是"能不能用LLM",而是"如何让LLM分析得准、出错时不毁信任"。
分类器两段式: 规则/小模型粗筛(便宜) + LLM精判(拿不准的)。分类是初始路由非死判决,非代码分支发现证据指向代码可回切。
两阶段LLM: 阶段1生成2-3个候选根因(发散), 阶段2强制回代码验证/否决(收敛)。全部否决则判低置信。对冲单轮"瞎猜"。
非代码也查代码: 缺文件必须查路径引用确认外部还是内置; 配置变更查读取点; 依赖故障查有无重试熔断。这是相对"只看代码"工具的差异化。
检索非单一RAG: grep精确(主力) + codegraph结构(关键-根因在调用链/数据流里) + 向量语义(补充) + 历史案例RAG(专长)。agent自主多轮迭代检索,类似Copilot范式。
置信度分流保护信任: 高置信自动定责+带方案; 中置信辅助triage给候选; 低置信仅通知标注AI未定位。宁可不说也不乱说——错误根因比没有根因更伤信任。先辅助姿态上线,攒准率再放开自动。
知识库全程辅助: 历史incident+修复+反馈回灌。方案被采纳是强信号,系统越用越准。
版本对齐: stack frame行号对应出错时release的commit版本,非main当前版本,否则行号对不上。
