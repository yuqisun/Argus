核心闭环：日志源 → 降噪 → 编排 → 根因分类 → (代码/非代码分支) → 方案+置信度 → 找最懂的人 → 邮件/IM 通知
本版不含：PR 自动化（后置）、SCM Provider 双平台适配（后置）
关键时序点：
1. 降噪在 LLM 之前，避免成本爆炸与噪音。
2. 根因类型先分类，代码 bug 才进 repo 检索，非代码走轻量分析。
3. 两阶段 LLM（候选到验证）显著提升根因准确率，对冲 OpenRCA 指出的难点。
4. 置信度贯穿全程，低置信走"辅助 triage"而非自动定责。
5. 反馈闭环：负责人对根因/方案的评价回灌 RAG 知识库，持续提升。
共享 Repo 检索：多手段配合。grep精确检索(主力) + codegraph结构检索(关键-调用图/依赖图/数据流)是核心; 向量语义检索和历史案例RAG是可选增强, 无向量库也能运行(grep+codegraph足够定位根因), 有则锦上添花(模糊匹配+历史对照)。agent 自主决定用哪种手段。
新增"缺文件/资源"分支：先查代码中的路径引用/资源加载点，确认该文件是外部依赖还是应内置，再判定是外部文件缺失还是代码缺失。
两阶段 LLM：阶段1 生成候选根因，阶段2 用实际代码上下文验证/否决，避免单轮"瞎猜"。
置信度分流：高置信自动定责+带方案；低置信退化为辅助 triage（给候选+人选，让人判定），保护信任。
核心定位: 把分析结果变成"给最懂代码的人的一份可执行修复请求"。产品姿态是"修复助手"不是"问责工具",团队接受度天差地别。
Owner 四级 fallback: CODEOWNERS(精确,团队维护) 到 OWNER文件(半结构化) 到 commit blame(出错方法主要贡献者,非最后一行作者) 到 README(LLM读兜底)。逐级 fallback 保证总能找到人。输出含来源标记("为什么找你")。
blame 改进: 不取出错行最后作者(可能只改了typo),取出错方法/类的主要贡献者。避免大重构自动修改导致的误判。
多平台身份映射: 邮箱作统一主键, GitHub username / ADO uniqueName 各自映射, 跨平台找到同一个人。
通知四要素: 根因摘要(位置+代码+为什么+类型) + 修改方案(代码bug给diff/非代码给建议动作) + 证据链(stack+变量+调用链+历史) + 置信度(高/中/低+依据)。让他收到能立即行动。
通知编排: 去重(同人短时合并) + 升级(高优N分钟未响应升oncall) + 静默时段(非紧急不凌晨发) + 频率控制(每人每日上限,超了聚合)。避免轰炸。
反馈闭环(飞轮): 四类反馈 - 根因准不准/方案能不能用/找错人没/已知问题。用途: 优化模型(采纳=强正信号) + 更新owner映射 + 加白名单 + 存知识库(可选RAG)。系统越用越准。
方案是起点非终点: 他最熟, 给起点比从零排查快。可改后采纳或否决自己修。反馈"改后采纳"是强优化信号。
成本漏斗：原始日志 10万条/分 - 多行聚合 - ERROR过滤 - 指纹聚合 - 时间窗去重 - 预过滤 - 进 LLM 仅 2-3 个/分。这一层做不好,整个系统因成本和噪音无法落地。
级别过滤可配置：默认 ERROR 及以上,但支持按服务/环境单独配置。有些团队业务异常打 WARN,有些服务 FATAL 才需关注,不能一刀切。配置项: 全局默认阈值 + 按服务覆盖 + 按环境覆盖(如 prod 比 staging 更宽松)。
指纹算法：不能 hash 整条日志(时间戳/host不同)。取 stack top-N 帧 + 异常类型 + message 模板,归一化动态段后 hash。偏粗会漏,偏细会炸,建议保守上线靠反馈调。
升级检测：静默期内若 host 数突增或堆栈变深,视为升级,突破 cooldown 重新触发分析。
预过滤三分流：白名单丢弃、规则/小模型判定非代码的直接分流、拿不准的才送 LLM。
保留原始样本：聚合后仍保留 1-2 条完整原始日志供 LLM 引用,否则 LLM 拿不到细节。
可回放：被过滤的异常可事后回查,避免过滤太狠漏真 bug 无法追溯。
核心职责:让重要异常优先分析,不重要延后或降级,LLM调用始终在可控预算内。无此层,洪峰时要么烧爆预算、要么雪崩卡死、要么关键事故被低优事件淹没。
优先级信号:影响范围+严重度+频率趋势+业务影响+新鲜度,加权打分分 P0-P3 四级。事件在队列等待时若影响扩大,优先级动态上调。
去抖 vs 降噪去重:降噪层去重基于 fingerprint 相同;编排层去抖针对同一根因的不同表象(发布 regression 触发多个不同 fingerprint),聚批后一次分析。P0 跳过去抖立即分析。
限流三维度:全局并发+QPS+token预算,按优先级分配配额。P0 占六成并发保证关键事故不被挤占。
降级关键原则:通知不能断。即使 LLM 挂了,至少把"有异常+找到的人+原始日志"通知出去让人工接手。分析能力可降级,告警能力必须高可用。
部分成功:根因分析成功但方案生成失败,根因仍通知,不浪费已有分析成果。
熔断:LLM 连续失败超阈值触发熔断,暂停调用一段时间,避免雪崩。
核心挑战: OpenRCA(ICLR2025)证明LLM定位代码根因准确率仍低。这层设计核心不是"能不能用LLM",而是"如何让LLM分析得准、出错时不毁信任"。
分类器两段式: 规则/小模型粗筛(便宜) + LLM精判(拿不准的)。分类是初始路由非死判决,非代码分支发现证据指向代码可回切。
两阶段LLM: 阶段1生成2-3个候选根因(发散), 阶段2强制回代码验证/否决(收敛)。全部否决则判低置信。对冲单轮"瞎猜"。
非代码也查代码: 缺文件必须查路径引用确认外部还是内置; 配置变更查读取点; 依赖故障查有无重试熔断。这是相对"只看代码"工具的差异化。
检索非单一RAG: grep精确(主力) + codegraph结构(关键-根因在调用链/数据流里)是核心; 向量语义(补充)和历史案例RAG(对照)是可选, 无向量库也能运行。agent自主多轮迭代检索,类似Copilot范式。
置信度分流保护信任: 高置信自动定责+带方案; 中置信辅助triage给候选; 低置信仅通知标注AI未定位。宁可不说也不乱说——错误根因比没有根因更伤信任。先辅助姿态上线,攒准率再放开自动。
知识库全程辅助: 历史incident+修复+反馈回灌。方案被采纳是强信号,系统越用越准。
版本对齐: stack frame行号对应出错时release的commit版本,非main当前版本,否则行号对不上。
扩展性: 采集降噪和通知服务无状态,按流量水平扩展。编排器和Agent有状态(限流/去抖),按分区扩展。Kafka分区按优先级隔离。
关键依赖: Redis(指纹缓存/限流/去重,必须高可用), Kafka(事件队列,持久化防丢), PostgreSQL(分析结果/配置,核心数据)。向量库可选, 无则grep+codegraph足够, 有则增强历史案例对照。
代码索引: tree-sitter AST/LSIF 建调用图+依赖图,按repo+commit版本索引,需定期重建。
配置中心: 服务/repo凭证/LLM endpoint/通知规则/白名单集中管理,热更新下发到各worker。
管理UI: Dashboard看分析历史/准确率/成本, 管理白名单/规则/owner映射, 配置通知策略。
高可用: 采集层和通知层多副本部署(通知不能断)。编排器单leader+备选(选主)。LLM不可用时降级为仅通知。
PostgreSQL: 核心业务数据。events存原始事件+指纹+处理状态(含可回放审计)。rca_results存分析结果供反馈和知识库。release_commits是版本对齐的关键——出错时查当前release对应的commit。
向量库(可选): 整个向量库是可选组件。pgvector(轻量,复用PG) 或 Milvus(大规模)。存历史incident的异常描述+根因+修复方案,用于RAG检索"上次类似的错怎么修的"。无向量库时系统仍可运行(grep+codegraph足够定位根因), 只是缺少历史案例对照能力, 准确率略低但冷启动更简单。
Redis: 高频读写状态。指纹缓存避免重复计算,限流/去抖/静默期状态都是短时TTL数据。必须高可用(挂了限流失效可能打爆LLM)。
代码索引: tree-sitter AST + LSIF/Scip调用图,按repo+commit版本索引。blame结果缓存(避免每次查git)。大repo索引重建耗时,需增量更新机制。
Kafka: 事件队列短期持久化(默认7天),服务重启不丢待分析事件。分区按优先级(P0/P1/P2分partition)。
数据生命周期: events保留90天(审计), rca_results永久(知识库), 向量库随反馈增长, Redis短TTL, 代码索引随commit增量更新。
脱敏必经: 生产日志含PII/密钥/token, 送LLM前必须脱敏。检测(正则+NER)识别敏感数据, 替换保留类型信息(EMAIL_1/API_KEY)便于LLM理解上下文但不泄露真实值。
凭证管理: GitHub App token/ADO PAT/LLM API key 存Vault/KMS, 自动轮换, 访问审计。绝不硬编码或存配置文件明文。
租户隔离: 按team/org隔离数据, 跨租户不可见。多团队共用一套Argus时, A团队看不到B团队的分析结果和代码。
LLM数据使用开关: 敏感repo可关闭"代码送LLM", 降级为仅通知(有异常+找人+日志, 无AI分析)。满足"代码不能送外部LLM"的合规要求。
私有部署选项: 支持私有部署LLM(vLLM/Ollama), 敏感场景数据不出内网。代价是模型能力可能弱于闭源模型, 准确率受影响。
操作追溯: 每次LLM调用记录输入输出hash(不存原文), 可审计"系统把什么送给了LLM"。
谁监控监控者: Argus给开发者监控异常, 自身也需要被监控。每个服务自带埋点, 指标采集到Prometheus, 日志到ELK, 追踪用OpenTelemetry。
四大指标维度: 成本(token/费用, 防超预算), 质量(准确率/采纳率, 核心价值指标), 性能(延迟/耗时, 影响响应速度), 可靠性(失败率/熔断/送达率, 影响可用性)。
质量指标靠反馈算: 根因准确率=反馈"准确"/总反馈。采纳率=方案被采纳/给方案总数。这是系统核心KPI, 直接反映LLM分析价值。
自监控告警: 成本超预算/LLM失败率飙升/通知送达失败/准确率下降——这些Argus自身的异常也要告警给Argus运维, 不能盲区。
事件ID贯穿全链路: 一个异常从采集到通知, 用同一event_id串联所有日志和trace, 方便排查"这个事件为什么没通知"或"这次分析为什么不准"。
Dashboard三视图: 运维大盘(实时健康), 质量大病(准确率趋势, 决定是否放开自动定责), 成本大盘(费用分解, 优化限流策略)。
多模型路由: 便宜模型做分类/粗筛(频繁调用), 强模型做根因/方案(关键调用)。重试用不同厂商模型避免同时故障。降低成本同时保证关键质量。
Prompt版本化管理: prompt模板存仓库, 每次改动可回滚。A/B测试不同prompt对准确率影响。变量注入异常/代码/历史案例。
上下文裁剪: tiktoken精确计量token。超限时按优先级裁剪: stack frame必留, 调用链次之, 历史案例取top-K, 日志只留异常前后N行。避免粗暴截断丢关键信息。
双层缓存: 指纹缓存(同fingerprint+同代码版本直接复用), 语义缓存(向量相似度判断相似问题)。大幅降低重复分析成本。
预算耗尽降级: token预算用完时, 不拒绝请求而是降级——只做分类+通知, 不做深度根因分析。保证通知不断, 分析延后。
冷启动: 知识库空时无历史案例可RAG, 准确率较低。建议预填已知问题库+Runbook bootstrap, 或上线初期全部走辅助triage不自动定责。
六步接入: 注册服务 到 授权仓库 到 配置Owner 到 配置通知 到 配置LLM策略 到 试运行上线。每步可回退修正。
Owner自动生成: 新团队没有CODEOWNERS时, Argus扫描git历史(谁提交最多)建议owner映射, 团队确认/修正。降低接入门槛。
Shadow模式试运行: 关键! 上线前先只分析不通知, 人工review准确率。达标后再开通知。避免上线即误报轰炸团队, 毁信任。
代码送LLM开关: 每个repo可单独配置是否允许代码送LLM。敏感repo关闭后降级为仅通知(有异常+找人+日志, 无AI分析)。
级别过滤按服务配置: 有些服务WARN就需关注, 有些FATAL才管。接入时按服务/环境设阈值, 不一刀切。
冷启动建议: 试运行期预填已知问题库+Runbook, 弥补知识库空时RAG无案例的劣势。上线初期全走辅助triage, 攒准率再放开自动。
